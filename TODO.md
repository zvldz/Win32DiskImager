# TODO

Work list inherited from upstream, with current status.

## Features

- **Compression support** — ✅ done in 2.2.0. `.gz` and `.xz` on Write + Verify (GUI + CLI), streaming decode with a multi-threaded xz decoder.
- **Reformat device** — not implemented. Brings an SD card back to a clean FAT32 layout after it's been written with a Linux / Pi image. Low priority: SDA Card Formatter and the Windows Disk Management snap-in already cover this.
- **GPT support for `Read Only Allocated Partitions`** — currently MBR-only. GPT-partitioned disks are detected (protective MBR with type 0xEE) and fall back to a full disk read with an info dialog. Native GPT support would parse the GPT primary header at LBA 1, walk partition entries (typically 128 × 128 bytes at the LBA from the header), find the highest-ending partition, and read both the start of the disk through that point AND the GPT backup header + partition entries at the disk tail (so the image is byte-exact restore-able with a valid backup GPT). ~150-200 lines including CRC validation. Worth doing if GPT-partitioned SD cards become common — currently rare on the SD / USB target media this tool focuses on.
- ~~**`--check-updates` for CLI**~~ — done in 2.2.3. Explicit one-shot version check via the GitHub releases API; never auto-runs (CLI stays quiet for scripting). Accepts both `check-updates` and `--check-updates`. Uses WinHTTP directly (no Qt) so the CLI binary keeps its existing dependency footprint.
- ~~**Show physical disks without a drive letter**~~ — done in 2.3.0. GUI Device dropdown and CLI `list` now enumerate `\\.\PhysicalDrive0..31` directly, filter by bus type / `GetDriveType` per letter, and map disks to their mounted letters. Bare disks (no FS Windows recognises) show as `Disk N — XXGB`. Lock + dismount covers every volume on the target disk. `WM_DEVICECHANGE` additionally subscribes to `GUID_DEVINTERFACE_DISK` so bare-disk insertion also triggers a refresh. CLI `--device` accepts both `1` (canonical, matches `list` output) and `E:` (legacy).
- **Adaptive xz decoder memory limit** — currently `XzImageReader` and CLI's `XzImageSource` set `lzma_mt.memlimit_threading = UINT64_MAX` and `threads = min(8, hardware_concurrency())`. For high-preset (`xz -9`) images each thread holds a ~675 MB dictionary, so 8-thread decode peaks at ~5 GB RAM regardless of system size — a system with 8 GB RAM will swap heavily or risk OOM. Fix: query `GlobalMemoryStatusEx().ullAvailPhys`, take a fraction (e.g. 25%) of available physical, clamp to a sane range (e.g. 256 MB ≤ X ≤ 4 GB), and set `memlimit_threading` to that. lzma_stream_decoder_mt will reduce the thread count to fit; if even one dictionary doesn't fit, it falls back to single-thread mode (~675 MB max). Apply identically in `src/xzimagereader.cpp` (GUI) and the `XzImageSource` block in `src/cli_main.cpp` (CLI). Optional `WDI_XZ_MEMLIMIT_MB` env var override for users who want to force a specific cap.
